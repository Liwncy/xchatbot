# MiMo TTS / AI唱歌插件设计稿（MVP）

> 面向当前 `xchatbot` 仓库的“AI 唱歌 / 语音合成”能力设计。
> 目标是在**尽量复用现有微信语音发送链路、AI 服务接入方式与插件框架**的前提下，落一个可迭代、风险可控的 `AI唱歌` 插件。

---

## 1. 目标与范围

### 1.1 目标

本设计稿希望实现一个文本触发的 `AI唱歌` 插件，支持：

1. 用户通过文本命令触发“唱歌”能力
2. 支持两种输入方式：
   - **主题生成**：输入一个主题/情绪/场景，由 AI 先生成一小段原创歌词，再调用 MiMo TTS 唱出来
   - **直给歌词**：用户直接提供原创短歌词，插件直接合成
3. 使用 **MiMo-V2.5-TTS** 的**预置音色**与**唱歌模式**完成演唱
4. 将结果以 **微信语音消息** 形式发出
5. 失败时支持合理降级（文本提示 / 可选外链）
6. 通过主人命令管理默认音色、风格、长度、开关等配置
7. 后续可自然扩展到：
   - 普通朗读（TTS）
   - 文本设计音色（VoiceDesign）
   - 音色复刻（VoiceClone）

### 1.2 MVP 非目标

MVP 暂不做：

- 背景伴奏 / 乐器编排 / 混音
- 真正“按某首流行歌曲原旋律复唱”
- 指定真实歌手音色模仿
- 音色复刻（VoiceClone）
- 公开开放的任意用户长文本/长歌曲生成
- Web 配置面板
- 流式边生成边发送（当前 MiMo 流式也未真正低延迟）

### 1.3 设计原则

1. **先做稳，再做炫**：先打通“文本命令 → MiMo TTS → 微信语音回复”的闭环
2. **先做预置音色**：MVP 只走 `mimo-v2.5-tts`
3. **先做原创短唱段**：避免版权、模仿、超长语音等复杂问题
4. **与现有仓库风格一致**：配置走 KV，入口走文本插件，回复走 `voice`

## 2. 当前仓库能力评估

当前仓库已有大量能力可以直接复用。

### 2.1 可直接复用的能力

- `src/plugins/types.ts`
  - 文本插件统一接口
- `src/plugins/index.ts`
  - 插件注册入口
- `src/types/message.ts`
  - 已有 `VoiceReply` 结构
- `src/wechat/index.ts`
  - 已支持 `voice` 类型回复发送
- `src/utils/silk-converter.ts`
  - 可将 `mp3/wav` 等非 SILK 音频转为微信可发的 SILK
- `src/utils/file-uploader.ts`
  - 可上传文件并拿到外链，适合语音发送失败时降级
- `src/plugins/ai/config.ts`
  - 已有成熟的 OpenAI-like 服务配置解析思路，可参考复用
- `src/plugins/common/ai-client.ts`
  - 已有通用 AI 文本调用方式，适合抽象出“音频版客户端”的风格

### 2.2 现有链路的关键意义

当前微信语音发送链路已经具备：

1. 插件返回 `VoiceReply`
2. `src/wechat/index.ts` 统一处理语音发送
3. 若回复音频不是 SILK，则调用 `normalizeVoiceForWechat(...)`
4. 转换后发送到微信网关
5. 如果发送失败，可自动降级成文本提示

这意味着：

> **MiMo TTS 插件只需要负责“拿到音频 + 填好 `VoiceReply`”，不用自己重新造一套微信发送链路。**

## 3. MiMo TTS 能力在本项目里的适配结论

根据你提供的文档，MiMo-V2.5-TTS 系列里，和“AI 唱歌”最契合的是：

- `mimo-v2.5-tts`

原因：

- 支持**预置音色**
- 支持**唱歌模式**（通过 `(唱歌)` / `(sing)` 标签）
- 使用门槛最低
- 不涉及音色设计和音色克隆的额外风险

### 3.1 MVP 只建议启用的模型

MVP 只启用：

- `mimo-v2.5-tts`

### 3.2 暂不建议在 MVP 中启用的模型

#### `mimo-v2.5-tts-voicedesign`

不建议首版启用，原因：

- 不支持唱歌模式
- 需要额外的描述提示设计
- 风格结果可控性还需要实测

#### `mimo-v2.5-tts-voiceclone`

首版明确不建议接入，原因：

- 不支持唱歌模式
- 涉及声音权利、授权和滥用风险
- 需要额外文件输入与管理逻辑

结论：

> **第一版聚焦“预置音色 + 原创短唱段 + 微信语音发送”。**

## 4. 用户交互设计

### 4.1 指令前缀

建议统一使用：

- `AI唱歌`

这样和现有 `AI对话` 管理命令风格一致，也便于后续扩展为完整的 TTS 插件。

### 4.2 普通用户指令

#### 查看帮助

```text
AI唱歌 帮助
```

#### 按主题生成一段原创唱词并唱出来

```text
AI唱歌 唱 夏夜操场、微风、暗恋但不敢表白
AI唱歌 唱 轻快一点，像在群里起哄唱两句
```

说明：

- 用户输入的是“主题 / 场景 / 情绪 / 风格偏好”
- 插件先让文本模型生成一小段原创歌词
- 再调用 MiMo TTS 的唱歌模式合成语音

#### 直接唱用户提供的原创短歌词

```text
AI唱歌 唱 <<
今晚的风吹过旧操场
你站在路灯下微微发光
我把没说出口的心事
都偷偷藏进了晚霞里
>>
```

说明：

- 仅建议短歌词
- 适合原创内容 / 内部测试内容

#### 试音（可选，建议保留）

```text
AI唱歌 试音 你好呀，我是小聪明儿，今天给你唱一小段。
```

说明：

- 不走唱歌模式，走普通 TTS
- 主要用于验证音色 / 发送链路

### 4.3 主人管理命令

#### 查看配置

```text
AI唱歌 配置
```

#### 开关插件

```text
AI唱歌 设置开关 开
AI唱歌 设置开关 关
```

#### 设置默认音色

```text
AI唱歌 设置音色 冰糖
AI唱歌 设置音色 茉莉
AI唱歌 设置音色 苏打
```

#### 设置默认唱歌风格标签

```text
AI唱歌 设置风格 活泼,轻快
AI唱歌 设置风格 温柔,怅然
```

说明：

- 最终会拼成类似：`(唱歌)(温柔)(怅然)`
- 也可只设置一个风格标签

#### 设置最大歌词长度

```text
AI唱歌 设置最大字数 120
```

#### 设置单段最长目标时长（近似控制）

```text
AI唱歌 设置单段秒数 25
```

说明：

- 实际不是精确秒控
- 用于约束生成歌词的字数和行数

### 4.4 推荐交互策略

建议首版默认：

- 普通群成员可直接使用 `AI唱歌 唱 ...`
- 但管理命令仅主人可操作
- 群聊不做“自动冒泡唱歌”
- 只在**明确文本命令**下执行

结论：

> `AI唱歌` 应当是**显式触发型插件**，不要做成自动唱歌。

## 5. 输入模式设计

插件在执行 `AI唱歌 唱 ...` 时，建议先判断输入属于哪一类。

### 5.1 模式 A：主题生成

判定特征：

- 输入为一行自然语言描述
- 不像歌词
- 没有明显多行结构

示例：

```text
AI唱歌 唱 深夜加班后终于下班回家的轻松感
```

处理方式：

1. 调用文本模型生成一段**原创短歌词**
2. 控制为 4~8 行、适合口语化演唱
3. 再调用 MiMo TTS 合成

### 5.2 模式 B：直给歌词

判定特征：

- 多行文本
- 或内容明显就是歌词结构

示例：

```text
AI唱歌 唱 <<
风从海边慢慢吹过来
把我的心事轻轻吹开
>>
```

处理方式：

1. 直接使用用户文本作为 `assistant.content`
2. 必要时做长度截断 / 清洗
3. 调用 MiMo TTS 合成

### 5.3 模式 C：普通试音/朗读

用于：

- 验证接口
- 验证音色
- 后续扩展为完整 TTS 插件

### 6.2 推荐插件名

- 代码名：`ai-sing`
- 对外命令前缀：`AI唱歌`

### 6.3 配置存储建议

建议新增独立 KV key：

- `plugins:ai-sing:config`

原因：

1. 与 `AI对话` 的聊天配置解耦
2. TTS 与文本对话是不同业务
3. 后续更方便扩展 TTS 专属配置

## 7. 配置模型设计

建议配置结构如下：

```json
{
  "enabled": true,
  "service": {
    "base_url": "https://token-plan-cn.xiaomimimo.com/v1/chat/completions",
    "model": "mimo-v2.5-tts",
    "api_key_secret": "MIMO_API_KEY"
  },
  "default_voice": "冰糖",
  "default_style_tags": ["温柔"],
  "max_lyrics_chars": 120,
  "target_segment_seconds": 25,
  "auto_upload_audio": false,
  "allow_theme_generate": true,
  "allow_user_direct_lyrics": true,
  "allow_group_use": true,
  "allow_private_use": true
}
```

### 7.1 字段说明

- `enabled`
  - 插件开关
- `service`
  - MiMo TTS 服务配置
- `default_voice`
  - 默认预置音色
- `default_style_tags`
  - 默认风格标签
- `max_lyrics_chars`
  - 单次歌词最大字数
- `target_segment_seconds`
  - 近似限制单段时长
- `auto_upload_audio`
  - 是否额外把音频上传为外链，便于失败降级
- `allow_theme_generate`
  - 是否允许主题转原创歌词
- `allow_user_direct_lyrics`
  - 是否允许用户直接提供歌词
- `allow_group_use` / `allow_private_use`
  - 控制可用场景

### 7.2 为什么不用直接复用 `AI对话` 配置

可以复用一部分结构思路，但**不建议直接共用同一个 KV 配置**。

原因：

- 对话插件关心的是“文本模型 / 记忆 / 提示词”
- 唱歌插件关心的是“音色 / 风格标签 / 字数 / 音频降级”
- 强行揉在一起会让管理命令变复杂

结论：

> **建议抽象出通用的 OpenAI-like 服务配置解析逻辑，而不是直接共用同一份业务配置。**

## 8. MiMo 请求构造设计

### 8.1 请求模型

MVP 固定使用：

- `mimo-v2.5-tts`

### 8.2 请求消息构造

根据 MiMo 文档：

- 目标合成文本必须放在 `role: assistant` 的消息中
- 风格控制说明可放在 `role: user` 中
- 唱歌模式必须在 `assistant.content` 开头加 `(唱歌)` / `(sing)` 标签

因此推荐构造：

```json
{
  "model": "mimo-v2.5-tts",
  "messages": [
    {
      "role": "user",
      "content": "请用自然、轻快、带一点青春感的方式演唱。"
    },
    {
      "role": "assistant",
      "content": "(唱歌)(温柔)(轻快)今晚的风吹过旧操场..."
    }
  ],
  "audio": {
    "format": "mp3",
    "voice": "冰糖"
  }
}
```

### 8.3 为什么建议音频格式先用 `mp3`

原因：

1. 更贴近现有随机语音插件的处理方式
2. 当前仓库已有 `mp3 -> silk` 转换链路
3. 和外部文件 URL 音频的兼容性通常更好

结论：

> **首版先固定请求 `mp3`，交给现有 `normalizeVoiceForWechat(...)` 转换即可。**

## 9. 主题生成歌词设计

### 9.1 是否需要先走文本模型

如果用户给的是“主题”，那么需要先生成歌词。

建议方案：

- 不直接让 MiMo 自己“边创作边唱”
- 而是先调用一个文本模型生成一段原创歌词
- 再把歌词交给 MiMo TTS 唱出来

这样好处：

1. 歌词更可控
2. 更容易做安全过滤
3. 更容易控制长度
4. 更方便后续加“风格模板”

### 9.2 歌词生成模型来源

有两种可选路径：

#### 方案 A：复用现有 AI 对话默认服务

优点：

- 实现最快
- 不必新增第二套文本服务配置

缺点：

- 和聊天插件耦合更深

#### 方案 B：在 `AI唱歌` 配置中单独加 `lyric_service`

优点：

- 更独立
- 后续更好调优

缺点：

- 首版配置略复杂

建议：

> **MVP 先用方案 A：默认复用现有文本 AI 服务生成原创歌词。**

### 9.3 歌词生成提示词建议

目标：

- 生成原创、简短、易唱、口语感强的小段歌词
- 避免太长、太书面、太像整首歌

建议提示方向：

```text
请根据用户给出的主题，写一段适合微信语音里清唱的原创短歌词：
1. 4 到 8 行
2. 总字数控制在 120 字内
3. 语言自然，容易唱，不要写成长篇诗歌
4. 不要引用现有歌曲歌词
5. 不要模仿真实歌手风格
6. 输出纯歌词，不要解释
```

## 10. 微信发送链路设计

### 10.1 结果处理流程

MiMo 返回后，插件需执行：

1. 取出 `completion.choices[0].message.audio.data`
2. 这是 base64 音频内容
3. 构造 `VoiceReply`
4. 指定：
   - `mediaId = base64音频`
   - `format = 3`（WAVE）
   - `duration = 估算值或默认值`
5. 由 `src/wechat/index.ts` 统一转 SILK 后发送

### 10.2 是否需要上传外链

建议做成可选配置：

- `auto_upload_audio = false`（默认）

如果开启，则：

1. 将生成的 `mp3` 上传到外部文件服务
2. 把返回 URL 填到 `VoiceReply.originalUrl`
3. 当发送失败时，可降级给出可打开的原始链接

### 10.3 发送失败降级建议

降级顺序建议：

1. 语音发送失败 → 文本提示
2. 若已上传外链 → 文本提示中附原始音频链接
3. 若没有外链 → 仅提示失败原因

## 11. 长文本与分段策略

### 11.1 为什么必须限制长度

即使 MiMo 能生成较长音频，微信语音消息场景也不适合过长内容。

问题主要有：

- 听感拖沓
- 失败概率提高
- 转换与发送成本增加
- 群聊体验变差

### 11.2 首版建议

首版建议：

- 单次目标控制在 **10~30 秒** 体验区间
- 通过字数和行数近似控制
- 默认只发**单段**

### 11.3 后续增强

如果歌词过长，可在后续版本支持：

- 自动分段
- 多条语音连续发送
- 在每段前加标题序号，如 `第 1 段 / 第 2 段`

建议：

> **MVP 先限制输入长度，不急着做自动多段。**

## 12. 安全、版权与风控建议

这一部分非常关键。

### 12.1 不建议支持的请求

MVP 应明确避免：

- “模仿某真实歌手声音唱”
- “把某首现有流行歌完整唱出来”
- “照着原版歌词直接唱整段”
- “复刻某公众人物声音”

### 12.2 建议的处理原则

#### 对“指定真实歌手音色”

回复为：

- 不直接模仿真实人物音色
- 可以改为“相近气质但非特定真人”的原创演绎

#### 对“唱某首已有歌曲”

回复为：

- 不直接输出完整原歌词
- 可以改成“同主题原创短唱段”

#### 对用户直接输入歌词

建议只允许：

- 短文本
- 明显原创内容
- 或内部测试用途

实现上可做简单风控：

- 超长歌词直接拒绝
- 命中高风险关键词时转为提示

结论：

> **首版最稳妥的产品定位是：原创短唱段生成器，不是点歌翻唱器。**


可正确读取与更新配置

4. 接口失败、转换失败、发送失败时：

- 有明确日志
- 有用户可理解的降级提示

5. 不应破坏现有：

- `AI对话`
- 其他文本插件
- 微信语音发送链路

---

## 15. 最终建议结论

如果目标是“先做一个能用、能发、能迭代的 AI 唱歌插件”，那么最优策略是：

1. **插件独立**，不要硬塞进 `AI对话`
2. **模型固定** 用 `mimo-v2.5-tts`
3. **只用预置音色**，先别碰 VoiceClone
4. **默认做原创短唱段**，不要做点歌翻唱
5. **输出固定走 `mp3 -> silk -> 微信 voice`**
6. **默认显式命令触发**，不要自动唱
7. **配置独立存 KV**，方便后续单独演进

一句话总结：

> **第一版把它做成“微信群/私聊里可调用的原创短唱段语音插件”，最稳、最符合当前仓库结构，也最容易快速落地。**
